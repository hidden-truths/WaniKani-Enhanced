# Regenerate / extend the JLPT word-list data

You are changing the vocabulary data that powers the 合格 JLPT tab's coverage lens, the
N3 gap-fill card flow, and the `jlptOf()` badges the 鰐蟹 WaniKani tab and card activation
stamp with. This skill keeps you from touching the wrong artifact (there are two, plus a
third server-side one) and from mangling the frequency ordering that the gap-fill silently
depends on.

The whole thing is a **generator + committed output**: one script reads a JLPT-level list
and a locally-downloaded JMdict, and writes the per-level `.js` chunks. You never hand-edit
the generated files; you re-run the generator.

## Before you start

Read the JLPT feature module first — it is the hub every consumer goes through:
`study-app/src/features/jlpt/data.js`. It owns the two lazy loaders (`ensureJlptMap`,
`ensureJlptWords`) and the sync fail-soft `jlptOf`. Skim the pure logic in
`study-app/src/core/jlpt.js` and the study-app JLPT bullets in
[study-app/CLAUDE.md](../../../study-app/CLAUDE.md) ("合格 JLPT state" + the gap-fill /
`jlptfill` dead-ends). For dev-loop / test mechanics defer to the `study-app-dev` skill;
for the server-side scoring artifact see `api-dev`.

**Know which of the three artifacts you're touching** — this is the #1 mistake:

| Artifact | Path | What it is | Regenerated by |
|---|---|---|---|
| Level **map** | `study-app/src/data/jlpt.js` | 7604-word N5–N1 map, compact `level: "w,w,w"` strings. Its own Vite chunk, dynamic-imported by `ensureJlptMap()`. Backs `jlptOf()` badges + coverage. | a python snippet (in this file's git history) from `wk-enhanced-api/data/jlpt-vocab.json` — see Traps |
| Per-level **entries** | `study-app/src/data/jlpt-words/<N5..N1>.js` | JMdict-enriched `[jp,read,mean,cat,type,trans]` tuples, **frequency-ordered**. The gap-fill card source, lazy-loaded per level by `ensureJlptWords(level)`. | `wk-enhanced-api/scripts/generate-jlpt-words.ts` (this skill's main job) |
| Server **scoring** | `wk-enhanced-api/data/jlpt-vocab.json` | 7604-entry `{headword: 1..5}` list. Feeds `scoreJlpt()` for userscript payload JLPT ceilings — NOT an app lens. | hand/external; also the INPUT to both artifacts above |

`jlpt-vocab.json` is the common source for all three: the map and the entries are both
**derived from it**. If a word is mis-leveled everywhere, fix it there and regenerate both.

## Regenerating the per-level entry chunks (the main task)

The generator enriches each headword in `jlpt-vocab.json` with a reading/gloss/POS from
JMdict and writes the five `<N>.js` files. Confirm the exact invocation against the script
header rather than trusting this page — but as of 2026-07 it is:

```bash
# 1. Download JMdict_e locally (NOT committed — EDRDG dictionary, licence-attributed in output):
curl -sL http://ftp.edrdg.org/pub/Nihongo/JMdict_e.gz | gunzip > /tmp/JMdict_e

# 2. From wk-enhanced-api/, run the generator (input path is the only arg):
cd wk-enhanced-api
bun scripts/generate-jlpt-words.ts /tmp/JMdict_e
```

It reads `wk-enhanced-api/data/jlpt-vocab.json` itself (relative to the script), matches
each headword against JMdict, and writes to `../study-app/src/data/jlpt-words/{N5,N4,N3,N2,N1}.js`.
Per level it logs `N3: 2069 words (N missed)`; missed headwords (JMdict had no matching
entry) are listed at the end — patch by hand or accept the gap. As of 2026-07 all 2069 N3
words matched (0 missed).

**The order IS the selection order.** Each file is sorted by JMdict priority markers
(`ichi1`/`news1`/`nfXX` → a frequency score in `priScore`), highest first. The gap-fill
batch (`selectGapBatch` in `core/jlpt.js`) walks the array top-down, so the ordering
literally decides which words a learner is offered first. Don't re-sort the output
alphabetically or by any other key — you'd silently change what gets studied.

## Extending a level (e.g. "add N2 coverage")

Good news: **the app side is already wired for all five levels.** Before planning any UI
work, verify this hasn't drifted:

- `ensureJlptWords`'s `WORD_LOADERS` in `data.js` already maps `N5..N1` → their chunks.
- `view.js` renders a level chip per `JLPT_LEVEL_ORDER` (`['N5','N4','N3','N2','N1']` from
  `core/jlpt.js`) as `data-jl-act="level"` buttons — the picker already offers every level.

So "add N2" usually means only: **ensure `jlpt-vocab.json` actually contains N2-leveled
headwords, then regenerate.** If the source list is missing a level's words, the generator
emits an empty/short `N2.js` and the coverage lens shows near-zero — that's a source-data
gap, not app wiring. The N3 grammar lens and the four-papers guidance in `view.js` are N3-
tuned regardless of the selected level (copy notes this); that's intentional, not a bug to
"fix" when a user picks N2.

Test: `study-app/test/jlpt-core.test.js` builds a `buildJlptMap` fixture per level and
pins coverage/gap/countdown math — extend those cases if you change the shape, not just the
data.

## Traps (with why)

- **The `jlpt.js` MAP has no committed generator — its header points at "the python snippet
  in the git history of this file".** As of 2026-07 there is no `.ts`/`.py` in the tree that
  emits `src/data/jlpt.js`; it's produced from `jlpt-vocab.json` by an ad-hoc script recorded
  only in that file's git history (`git log -p -- study-app/src/data/jlpt.js`). If you need to
  rebuild the map (e.g. after editing `jlpt-vocab.json`), recover that snippet from history or
  re-derive it — do NOT hand-edit the 7604-word strings. Consider promoting it to a committed
  script if you touch this more than once (a reasonable roadmap item; see the `roadmap` skill).

- **`jlptfill` is the gap-fill SOURCE token, NOT `jlpt`.** Gap-fill cards created by
  `addJlptWords` (`features/jlpt/activate.js`) carry `jlptfill:true` + `added:'YYYY-MM-DD'`.
  `jlpt` is the LEVEL facet (`.chip.jlpt`, the N5–N1 segmented control) — a source chip named
  `jlpt` would collide with its wiring. The `added` day-stamp is the checklist quota's live
  signal (the 語 auto-row), so don't drop it. Dedup is a **headword skip** (a word already in
  the deck is not re-added), the songs-style path — not a Minna-style overlay.

- **Don't "fix" the server's fail-open JLPT scoring with suffix heuristics.**
  `scoreJlpt` (`wk-enhanced-api/src/lib/jlpt.ts`) scores IK sentence tokens against
  `jlpt-vocab.json`; it deliberately treats unknown/conjugated tokens as a skip and returns 0
  ("no identifiable tokens" → passes any ceiling) rather than guessing. IK returns surface
  forms (`食べた`) and the list has dictionary forms (`食べる`), so mismatch is high by design.
  Stem-stripping without a real morphological analyzer creates more false matches than it
  solves — this is a pinned dead-end in both the userscript and `wk-enhanced-api/CLAUDE.md`.
  It's a different artifact/concern from the app lenses; touching one doesn't require the other.

- **Regenerating changes a big committed file — commit it as data.** The `<N>.js` chunks are
  hundreds of KB. One logical change → one commit (repo convention). If only glosses/order
  shifted because you re-ran against a newer JMdict, say so in the message; that's expected
  churn, not a bug.

- **The map and entries can disagree on membership.** The coverage lens counts via the MAP
  (`jlptLevelTotal` / `deckJlptCoverage` over `jlpt.js`), while gap-fill offers ENTRIES
  (`jlpt-words/<N>.js`). Both derive from `jlpt-vocab.json`, but the entries drop any headword
  JMdict couldn't match (the "missed" list). So a level's gap-fill pool can be slightly smaller
  than its coverage denominator — that's the JMdict-miss gap, expected, not a counting bug.
