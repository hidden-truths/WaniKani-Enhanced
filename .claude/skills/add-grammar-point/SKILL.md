# Add or edit N3 grammar-catalog content

You are authoring exam content for the project's single user (JLPT N3, exam default
2026-12-06). Grammar was the zero-coverage exam paper until wave 1 of the exam push shipped
the 81-point catalog (2026-07-02). The pipeline is: a durable **id registry** + per-point
**content JSON** → a **build script** that validates and emits a generated catalog module →
cloze SRS cards, a JLPT-tab coverage lens, and Browse detail that all render **by id lookup**
→ a server **seed pass** that mirrors the example sentences into the sentence store for
tap-a-word tokens and TTS. Getting the ids and the cloze `blank` right is what this skill
protects; everything else is mechanical.

## Before you start

1. **Read `study-app/tools/grammar-n3/CONTENT_GUIDE.md` in full.** It is the binding content
   contract (registry shape, per-point JSON shape, example rules, workflow). This skill adds
   the surrounding pipeline and traps; it does not replace the guide.
2. **Check whether the point already exists** — the obvious N3 patterns are mostly taken
   (81/81 registry entries have content as of 2026-07; e.g. 〜に違いない is already
   `ni-chigainai`):
   ```bash
   grep -i "chigai" study-app/tools/grammar-n3/points.json   # search by romaji fragment
   grep "違いない" study-app/tools/grammar-n3/points.json     # or by label
   ```
   If the id exists you are **editing** `content/<id>.json`; only a genuinely new pattern
   adds a registry entry.
3. Orient in the module map if you'll touch app code (`study-app/CLAUDE.md` grammar bullets;
   the `study-app-dev` skill). Pure content work needs no app-code changes at all.

## The two generated artifacts (never hand-edit either)

| Artifact | What it is | Regenerated by |
|---|---|---|
| `study-app/src/data/grammar-n3.js` | The **curriculum**: 81 N3 points × explanation/formation/3–5 cloze examples (as of 2026-07). Exports `GRAMMAR_N3`; lazy-loaded as its own Vite chunk. | `node tools/grammar-n3/build.mjs` |
| `study-app/src/data/grammar.json` | The **GiNZA tagger catalog**: 38 N5/N4 detector points (as of 2026-07) that label sentence grammar tags. | `sentence-nlp/patterns.py` (a different pipeline — see the grammar-filter dead-end in `study-app/CLAUDE.md`) |

Separate artifacts, **one shared id vocabulary**. `build.mjs` cross-checks every registry id
against the tagger catalog: an **exact collision is a build error** (no wave-1 N3 point
overlaps the N5/N4 tagger set; if a genuinely-identical pattern ever needs to share an id,
the build's comment describes loosening via an explicit manifest flag — that flag does NOT
exist today, don't invent it). A **prefix near-collision only warns** (`hazu-ga-nai` vs
tagger `hazu` is fine — different points).

## Durable ids are sacred

**Never rename a shipped id.** Reasons, so you can defend the rule:

- Activated grammar cards store `grammarId` and render their content **by lookup**
  (`grammarPointOf(id)`), never by embedded copy. Rename an id and every existing card
  orphans — label renders but explanation/formation/cloze silently vanish.
- Wave-2 MCQ banks (`grammar-mcq-drills` in `ROADMAP.html`) will key on the same ids.
- Server rows are keyed `gp-<pointId>-<ordinal>` — a rename strands seeded rows.

The flip side is the payoff: **content fixes reach existing cards without re-activation**.
Fixing a typo in an example is: edit JSON → rebuild → done. No migration, no card touch.

## Authoring procedure

Work from `study-app/` for all commands.

**1. Registry entry (new point only).** Append to `tools/grammar-n3/points.json`:
`{ "id", "label", "read", "mean" }` — id is a durable kebab-case slug (validated
`^[a-z0-9]+(-[a-z0-9]+)*$`), `label` the display pattern (`〜に違いない`), `read` kana,
`mean` a short English gloss. Pick the id as if it must survive forever, because it must.

**2. Content file** `tools/grammar-n3/content/<id>.json` (filename must equal the id):

```json
{
  "id": "<the manifest id>",
  "explanation": "2–4 learner-facing sentences: core meaning, nuance, contrasts/pitfalls.",
  "formation": "V-dict / N の + パターン style attachment note.",
  "examples": [ { "jp": "…", "en": "…", "blank": "…" } ]
}
```

What the **build enforces** (it will catch you): id in the registry; filename matches;
explanation ≥ 40 chars; formation non-empty; 3–5 examples; every `jp` is clean ruby (only
well-formed `<ruby>漢字<rt>かんじ</rt></ruby>`, no nesting, no other tags); `blank` found in
`plainText(jp)` (error if absent); warnings when `blank` occurs more than once (first
occurrence gets clozed) or intersects a ruby segment (the cloze would swallow furigana).

What **you must judge** (the build cannot): natural everyday Japanese at N3 level, 15–35
chars, the pattern present in every sentence, difficulty escalating across the set (aim for
4 examples), okurigana OUTSIDE the ruby (`<ruby>泳<rt>およ</rt></ruby>げる`, never
`<ruby>泳げる<rt>およげる</rt></ruby>`), `en` a natural translation not word-by-word,
vocabulary ≤ N3 where possible, and `blank` = the exact **kana pattern span**
(`ようになった`, `わけにはいかない`) — kana spans dodge the ruby-intersection warning by
construction.

**3. Build:**
```bash
node tools/grammar-n3/build.mjs
# success → wrote src/data/grammar-n3.js: <N>/<N> points (<M> examples)
#           (81/81, 322 examples before your change, as of 2026-07)
# errors  → listed, exit 1, generated file NOT written
```
Only registry ids that HAVE a content file are emitted; ids without content are listed as
`missing content (…)` so the catalog can grow incrementally. Treat warnings as review items,
not noise — a multiple-occurrence blank usually means you picked too short a span.

**4. Test:**
```bash
bun run test test/grammar-core.test.js   # 9 tests as of 2026-07; catalog invariants run
bun run test                             # full suite before committing
```
`test/grammar-core.test.js` pins the invariants over the REAL generated module (unique ids,
complete fields, 3–5 clean-ruby examples, resolvable blanks) — a bad regen fails loudly here
even if you skipped the build output.

**5. Commit + flag for proofread.** Stage exactly the three paths (registry, content file(s),
generated module) — never `git add -A`; parallel agents may share this tree. One logical
change, one commit, house style `study-app: …`. Then follow the `land-a-change` skill.
**Model-generated Japanese is not exam-trusted until the user has proofread it** — the
pipeline is explicitly model-drafted → build-validated → human-proofread (`content-proofread`
is a standing record in `ROADMAP.html`). Say in your summary which points you added/changed
and that they await the user's proofread. As of 2026-07-06 the user's proofread of the
original 81-point batch is itself still pending.

## Where the content surfaces (downstream map)

- **Catalog loader** — `study-app/src/features/grammar/data.js`: `ensureGrammarPoints()`
  dynamic-imports the generated chunk; `grammarPointOf(id)` is sync and fails soft to `null`
  (a card whose chunk hasn't landed renders label/mean without cloze for one paint).
- **Activation** — `features/grammar/activate.js` `activateGrammarPoints()`: grammarId-dedup
  (`grammarDeckIndex`), monotonic ranks, builds cards via the pure `buildGrammarCard`
  (`cat:'grammar'`, tags `['文法']`, `grammar:true`, `grammarId`). Triggered from the JLPT
  tab's ACTIONS (`gp-add`, `gp-add-all`, `go-grammar-drill` in `features/jlpt/view.js`).
- **Flashcard cloze branch** — `features/flashcard.js`: prompt face renders the example with
  the blank as an opaque ＿-gap; answer face reveals it marked, plus explanation/formation
  looked up by grammarId. `pickGrammarExample(examples, attempts)` rotates deterministically,
  so each review shows the next sentence.
- **Browse detail** — `features/browse.js` renders the point's full content by grammarId.
- **JLPT-tab lens** — `core/grammar.js` `grammarCoverage()` (new/added/learning/solid per
  point) feeds the readiness lens and the pacing coach; `grammarReviewedToday()` drives the
  法 checklist row (conditionally auto once grammar cards exist).
- **Tap-a-word tokens** — progressive enhancement via a read-through resource
  (`jpverbs_grammar_cache`, `GET /v1/sentences?ownerType=grammar_point&annotate=1`), keyed
  `<pointId>:<ordinal ?? 0>`. The `?? 0` default is load-bearing: the wire compacts a falsy
  ordinal away, so without it every point's FIRST example loses its tokens. Missing tokens
  are fine — the answer face falls back to plain ruby.

The pure helpers all live in `study-app/src/core/grammar.js` (DOM-free, tested).

## Server side: seed, TTS, tokens

The generated catalog is the **runtime content source** — the app's cloze drill works with
zero server involvement. The store rows exist only for the NLP token layer and TTS pre-gen.

- **Seed** — `wk-enhanced-api/scripts/seed-sentences.ts` Pass 5 upserts every catalog example
  as a PUBLIC row: `ext_id` = `gp-<pointId>-<ordinal>`, `source='grammar'`,
  `tags {grammar:[pointId]}`, link `{owner_type:'grammar_point', owner_id:pointId, ordinal}`.
  Idempotent by ext_id (a re-run refreshes text/furigana/translation in place). 322 rows for
  the 81-point batch. Run from `wk-enhanced-api/` with `bun scripts/seed-sentences.ts`
  (dev DB via `.env`; prod runs on the droplet — the `deploy-prod` skill has the compose-run
  pattern and its ENV_FILE/DATA_DIR gotcha).
- **TTS** — `wk-enhanced-api/scripts/collectTtsTexts.ts` enumerates every grammar example
  (`plainText(ex.jp)`), so `generate-tts.ts` pre-renders Siri clips and the variants manifest
  covers them. Playback does NOT block on this: the answer-face speaker button hits
  `/v1/audio/tts`, which lazily falls back to Google TTS for any un-pre-generated text.
  Pre-gen is a voice-quality upgrade, not a prerequisite.
- **GiNZA tokens lag by design.** Order: seed rows → offline parse on the Mac
  (`sentence-nlp/`, no Python on the droplet) → committed `data/annotations.json` →
  `seed-annotations.ts` (resolves by content hash). New/changed examples render plain ruby
  until that batch re-runs. Fail-soft; set expectations, don't chase it as a bug.

**Prod state, probed 2026-07-08: wave 1 IS deployed and seeded** —
```bash
curl -s "https://api.wkenhanced.dev/v1/sentences?ownerType=grammar_point"
```
returns 322 rows. Re-run that probe before reasoning about prod rather than trusting this
line: a validation error naming only `selftalk|card` = API not deployed; `{"sentences":[]}` =
API deployed but seed not run; rows = deployed + seeded. New content reaches prod only via the
`deploy-prod` skill: rebuild the web container (new bundle chunk), rebuild the API container if
server code changed, re-run `seed-sentences.ts` on the droplet, optionally TTS pre-gen,
eventual NLP re-parse. (The grammar/Minna sentences still fall back to Google TTS — no Siri
clips pre-generated; roadmap `infra-siri-voices-grammar-minna`.)

## Traps

- **The blank must sit in kana, outside every ruby base.** A blank overlapping a
  `<ruby>` segment makes the cloze swallow the furigana (build warns). Prefer the kana
  pattern span; if the pattern's only occurrence includes kanji, rephrase the sentence.
- **A too-short blank hits the multiple-occurrence warning** (e.g. blanking `ように` in a
  sentence that also contains ような). Only the first occurrence is clozed — usually wrong.
  Blank the full pattern span or rewrite.
- **Okurigana inside the ruby base breaks furigana rendering conventions** —
  `<ruby>泳<rt>およ</rt></ruby>げる`, never `<ruby>泳げる<rt>およげる</rt></ruby>`. The build's
  clean-ruby check does not catch this (it's well-formed markup, just wrong); the human
  proofread does.
- **Editing an example's `jp` text then re-seeding leaves any previously-attached GiNZA
  annotation stale** — `upsertPublicSentence` replaces text/hash in place but does not clear
  `sentence_annotation`, so old token offsets can misalign against the new text until the
  next offline parse re-annotates. Tolerable window (grammar rows may have no annotations at
  all yet as of 2026-07 — prod isn't even seeded); just don't debug it as an overlay bug.
- **Removing or reordering examples shifts ordinals.** Re-seed updates each
  `gp-<id>-<ordinal>` row in place; a removed trailing example leaves an orphan row nothing
  deletes. Harmless (the catalog, not the store, is the content source, and orphan tokens are
  never looked up) — but prefer appending over reordering to keep ordinals stable.
- **Don't "fix" a same-pattern id collision by renaming the shipped id** — see the durable-id
  section. And don't merge the two generated artifacts; they deliberately share only the id
  vocabulary (`study-app/CLAUDE.md` has the dead-end).
- **`grammar`/`grammarId` are machine-set card fields** carried through `saveVerb`'s explicit
  copy list on edit (`features/custom-cards.js`). If you ever add a new machine-set field to
  `buildGrammarCard`, add it to that carry-through list or editing a card in the modal
  orphans it from its content.
- **Extending to N4/N2 is a new pipeline instance, not a tweak** — new registry + content dir
  + generated chunk + lens wiring. The shape is level-agnostic, but scope it as a
  `ROADMAP.html` record first (the `roadmap` skill), and check `content-gap-audit` for
  whether it beats other exam work. Wave-2/3 drills (`grammar-mcq-drills`,
  `jlpt-vocab-drills`) are already-decided roadmap records; reading/listening practice
  deliberately stays external to the app (the hybrid decision) — don't propose tabs for them.
